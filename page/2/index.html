4-18学习总结

4-18号总结

GIT打包

GIT打包是一种线下共享代码的方式,当有时候由于权限或者网络等原因无法在线拉取代码的时候,就可以使用git打包这种方式.打包的主要步骤如下.
  • git bundle create repo.bundle HEAD master. 打包master分支,包含最近的一次提交
  • git fetch/pull repo.bundle master:local-master. 从repo.bundle中抽取出master分支合并到被合并的local-master分支.
还有一些额外的命令帮助打包:
  • git log –oneline :查看历史提交记录

EventBus初探

EventBus学习记录

  • EventBus的使用的步骤

    1. EventBus在JCenter和Maven Central中都是可用的,所以可以直接在gradle构建文件中进行添加compile'org.greenrobot:eventbus:3.0.0'

    2. 定义一个event类,例如 MessageEvent

    3. 在subscriber（订阅者）中声明处理Event的回调函数:例如处理MessageEvent对象就声明一个onMessageEvent()回调函数,并在函数前面加上@subscribe注解.

    4. 在activity或service等组件的生命周期回调函数上加入注册和解注册的逻辑. EventBUs.getDefault().register(this);
      EventBus.getDefault().unregister(this);
    5. 在代码的任何位置post event对象.
      EventBus.getDefault.post(new MessageEvent())

EventBus的几种处理模式:

  • postThread:默认模式,在post所在的线程直接调用订阅者的事件响应方法,不论该线程为什么线程.
  • MainThread: 在主线程中响应,如果发布线程即为主线程,则直接调用订阅者的事件响应函数,否则通过主线程的Handle post回主线程中执行.
  • AsyncThrad: EventBus在初始化的时候会新建一个 CachedThreadPool exuecteService,在 async模式下会使用该线程池来处理event.
  • BackGroundThread:同样使用线程池来对event进行处理,但与async模式不同的地方在于:在background模式下每次只有一个线程能执行任务,而async模式下所有线程都是并发执行的.(源码分析如下)

EventBus架构有什么好处:

  • 实现了消息发送者和接受者间逻辑的解耦
  • 提供了对消息事件的多线程处理.
  • 还有很多,暂时没有体会到………

上午

把系统升级的bug解出来了,发现其实代码中早已经有处理的逻辑了...
以后看代码一定要注意边看边画流程图或者作记录,否则效果会很差.

下午:

1.   新了解了一个类
__RemoteCallbackList__

处理持有一系列远程接口的任务,最典型的应用就是为执行从Service到客户端的回调.
大体上,会跟踪一系列注册的回调,通过它们的潜在的唯一的IBinder对象来识别这些接口.为每一个注册的接口关联一个DeathRecipient对象，这样当接口所在的进程结束后接口能被及时清除.
执行对接口列表的加锁来处理多线程调用请求,并提供一个线程安全的方法在没有持有锁的情况下反复迭代接口列表的快照.

这个类的主要时为了方便多进程中service与客户端的通信.

  1. 对notification有了一些新了解:
  • 当设置了onGoing这个选项后通知栏中的通知便不能被用户划掉,只能由系统或者代码来取消.
  • 可以通过Notification.Builder.addAction(id,title,intent)这个方法来为通知添加额外的点击事件
